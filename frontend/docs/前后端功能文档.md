# 限员管控系统前后端功能文档



## 1. 系统总体说明

系统由门禁事件后端和大屏前端两部分组成：

- 后端负责：
  
  - 接入门禁设备 SDK
  - 采集门禁刷脸/进出事件
  - 订阅人数统计、区域报警、异常报警等 MQTT 消息
  - 执行限员门控和异常门控
  - 提供聚合 Web API 给前端读取

- 前端负责：
  
  - 展示实时人数、报警、停留人员、异常消息
- 支持限制人数手动调整
- 支持停留报警时间手动调整
- 支持区域报警处理、异常消息处理关闭
- 支持异常弹窗与语音/录音播报

## 2. 后端功能说明

### 2.1 门禁设备接入

后端支持从 `DeviceConfig.xml` 读取多台门禁设备配置，并启动多门禁事件服务。

已实现能力：

- 多设备登录与布防
- 实时接收门禁刷脸事件
- 记录门编号、设备 IP、人员信息、进出方向
- 支持按设备方向区分进门与出门

### 2.2 员工信息同步

系统启动时会从配置的 `ApiBaseUrl/api/employee` 拉取员工信息，并建立本地匹配关系。

匹配字段包括：

- `employeeId`
- `employeeNo`
- `cardNo`
- `name`

用途：

- 将门禁抓拍记录映射为姓名、部门、岗位等展示信息

### 2.3 Web 聚合接口

后端内置 Web 服务，默认端口来自 `DeviceConfig.xml` 的 `WebPort` 配置，常用为 `8081`。

主要接口如下：

- `GET /api/dashboard`
  
  - 返回前端大屏所需聚合数据
  - 包含指标、报警、最近进出记录、停留人员、区域报警、异常消息

- `GET /api/limit-count`
  
  - 读取当前限制人数

- `POST /api/limit-count`
  
  - 更新限制人数

- `POST /api/abnormal/close`
  
  - 手工关闭一条异常消息

- `GET /events`
  
  - 返回原始门禁事件列表

- `GET /config`
  
  - 返回当前 `DeviceConfig.xml`

- `GET /config/edit`
  
  - 在线编辑设备配置

- `GET /images`
  
  - 查看抓拍图片列表

- `GET /health`
  
  - 健康检查接口

说明：

- 后端已配置 CORS，便于前端跨域访问

### 2.4 MQTT 订阅能力

后端当前支持以下 MQTT 订阅：

- `people_count`
  
  - 人数统计消息
  - 用于更新当前区域人数

- `area_alert`
  
  - 区域报警消息
  - 用于触发区域报警状态和前端大屏报警提示

- `abnormal`
  
  - 异常消息
  - 当前重点处理“尾随闯入”业务

### 2.5 人数统计与限员控制

后端支持基于人数统计或门禁事件推算当前在场人数，并执行限员门控。

涉及配置项：

- `LimitCount`
- `ExitGraceSeconds`
- `StayWarningMinutes`
- `RecentRecordCount`
- `CapacityWarningRatio`

已实现规则：

- 当前人数达到限制人数时：
  
  - 进门设备切换为禁入状态
  - 出门设备保持可用

- 满员状态下检测到出门刷脸时：
  
  - 临时解除限员门禁限制
  - 等待人数下降后恢复限制

### 2.6 尾随闯入异常门控规则

后端已实现 abnormal 订阅中的“尾随闯入”专用规则。

异常消息示例：

```json
{
  "type": "尾随闯入",
  "time": "2026-04-06 16:51:36",
  "zoneName": "门禁165",
  "conditionMet": 1
}
```

处理规则如下：

- `type = 尾随闯入`
- `zoneName`
  - 作为门名/区域名匹配依据
- `conditionMet = 1`
  - 设置对应门为常闭状态
- `conditionMet = 0`
  - 恢复门禁正常受控状态
  - 同时自动将对应异常消息标记为已处理

### 2.7 异常消息状态管理

后端维护异常消息列表，支持：

- 新消息入队
- 保存原始 payload
- 标记“待处理/已处理”
- 手工关闭异常
- `conditionMet=0` 自动关闭异常

当前规则补充说明：

- 同一条异常再次收到时，会重新进入“待处理”
- 因此前端会重新弹窗

## 3. 前端功能说明

### 3.1 页面概览

前端为单页大屏，看板主要展示：

- 核心指标
  
  - 进场人数
  - 出场人数
  - 限制人数
  - 区域内停留人员

- 活动报警

- 区域报警横幅

- 异常消息弹窗

- 最近进出记录

- 停留人员列表

- 实时时钟与状态提示

### 3.2 数据读取方式

前端不直接订阅 MQTT，而是通过轮询后端接口获取状态。

当前方式：

- 默认轮询 `GET /api/dashboard`
- 轮询周期：约 3 秒

前端 API 地址规则：

- 若配置了 `VITE_API_BASE_URL`，则优先使用该地址
- 未配置时，默认访问 `http://当前页面主机:8081`

### 3.3 限制人数管理

前端支持手动调整限制人数：

- 输入限制人数
- 点击“应用”调用后端 `POST /api/limit-count`
- 点击“恢复接口值”重新读取后端当前值

### 3.4 停留报警时间管理

前端支持手动调整停留报警时间：

- 输入停留报警分钟数
- 点击“应用”后先读取后端 `GET /config`
- 前端更新 XML 中的 `StayWarningMinutes`
- 再调用后端 `POST /config` 保存配置

### 3.5 区域报警展示与处理

前端支持区域报警：

- 区域报警激活时显示红色告警横幅
- 显示区域名、检测人数、时间
- 支持手工点击“处理”
- 处理后停止区域报警循环播报

### 3.6 异常消息展示与处理

前端支持 abnormal 异常弹窗：

- 仅显示当前待处理的异常消息
- 支持手工点击“处理关闭”
- 关闭后调用后端 `/api/abnormal/close`
- 后端再次把同一条异常置为待处理时，前端会再次弹出，无需刷新

### 3.7 语音与录音播报

前端支持三种播报方式：

- 浏览器语音播报（SpeechSynthesis）
- 录音文件播报（Audio）
- 蜂鸣提示音（最后兜底）

对于触屏/平板设备，当前优先启用录音播报。

### 3.8 abnormal 尾随闯入播报规则

前端已为 abnormal 的“尾随闯入”单独实现播报规则。

当前规则：

- 仅当以下条件同时满足时才播报：
  
  - `type = 尾随闯入`
  - `conditionMet = 1`

- 播报内容为固定尾随提示音

- 当 `conditionMet = 0` 时：
  
  - 不播报
  - 自动取消当前弹窗
  - 自动停止当前播报

### 3.9 尾随闯入专用音频文件

前端已配置尾随闯入专用音频通道。

当前播放顺序：

1. `public/voice-prompts/abnormal-tailgating.mp3`
2. `public/voice-prompts/abnormal-tailgating.wav`
3. `public/voice-prompts/abnormal-message.wav`
4. 蜂鸣声

当前已生成专用 WAV 文件：

- `D:\soft\personnumber\newdemo\public\voice-prompts\abnormal-tailgating.wav`

当前该固定音频的播报内容为：

> 请保持安全距离，请在警戒线外有序排队，依次刷脸通行。

说明：

- 固定音频文件无法动态读出 `zoneName`
- 如果后续希望播报“门禁165请保持安全距离……”这类带门名的动态内容，需要改为动态 TTS 方案

## 4. 前后端联动链路

### 4.1 正常门禁链路

1. 门禁设备上报刷脸事件
2. 后端接收并写入事件存储
3. 后端聚合 `/api/dashboard`
4. 前端轮询接口并刷新页面

### 4.2 尾随闯入链路

1. MQTT 推送 `abnormal` 消息
2. 后端解析 `type / zoneName / conditionMet`
3. 后端写入异常状态
4. 若为 `conditionMet=1`
   - 对应门切换为常闭
   - 前端下一次轮询看到待处理异常并弹窗
   - 前端播放尾随闯入专用语音/录音
5. 若为 `conditionMet=0`
   - 后端恢复门禁正常受控
   - 后端自动将异常标记为已处理
   - 前端自动关闭弹窗并停止播报

## 5. 配置说明

### 5.1 后端主要配置

配置文件：`DeviceConfig.xml`

常用配置项：

- `WebPort`
- `LimitCount`
- `RecentRecordCount`
- `StayWarningMinutes`
- `ExitGraceSeconds`
- `CapacityWarningRatio`
- `AreaAlertMqttEnabled`
- `AreaAlertMqttHost`
- `AreaAlertMqttPort`
- `AreaAlertMqttTopic`
- `AbnormalMqttEnabled`
- `AbnormalMqttHost`
- `AbnormalMqttPort`
- `AbnormalMqttTopic`
- `PeopleCountMqttEnabled`
- `PeopleCountMqttHost`
- `PeopleCountMqttPort`
- `PeopleCountMqttTopic`

### 5.2 前端主要配置

前端环境变量：

- `VITE_API_BASE_URL`

未配置时，默认使用：

- `http://当前主机:8081`

## 6. 运行说明

### 6.1 后端启动

1. 启动 `ACSEventConsole.exe`
2. 确认设备登录成功
3. 确认 MQTT 订阅成功
4. 确认 Web 服务成功绑定到 `8081`

### 6.2 前端启动

开发模式：

```bash
npm run dev
```

生产构建：

```bash
npm run build
```

## 7. 当前已实现的业务结论

当前系统已经具备以下完整闭环能力：

- 门禁事件采集
- 员工信息映射
- 人数统计展示
- 限员门控控制
- 区域报警展示与处理
- abnormal 异常弹窗展示
- 尾随闯入门控联动
- 尾随闯入专用语音/录音播报
- `conditionMet=0` 自动处理并自动取消播报

- 增加异常历史列表页
- 增加异常处理日志与审计记录
- 增加权限控制和角色管理
